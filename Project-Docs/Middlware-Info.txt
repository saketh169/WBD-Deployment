================================================================================
                    MIDDLEWARE TYPES DOCUMENTATION
                    NutriConnect WBD Backend Project
================================================================================

This document categorizes all middlewares used in the project across 5 types:
1. Built-in Middlewares
2. Custom Middlewares
3. 3rd-party Middlewares
4. Application-level Middlewares
5. Router-level Middlewares

================================================================================
1. BUILT-IN MIDDLEWARES (Node.js Native)
================================================================================

Built-in middlewares are part of Node.js Express framework without external packages.

1. express.json()
   - Purpose: Parse incoming JSON request bodies
   - Location: server.js (line 49)
   - Usage: app.use(express.json({ limit: '10mb' }))
   - Config: Limits body size to 10MB
   - Applied: Globally to all routes
   - Status: APPLICATION-LEVEL

2. express.urlencoded()
   - Purpose: Parse URL-encoded request bodies (form submissions)
   - Location: server.js (line 50)
   - Usage: app.use(express.urlencoded({ limit: '10mb', extended: true }))
   - Config: Limits body to 10MB, extended parsing enabled
   - Applied: Globally to all routes
   - Status: APPLICATION-LEVEL

3. express.static()
   - Purpose: Serve static files (images, documents)
   - Location: server.js (line 56)
   - Usage: app.use('/uploads', express.static('uploads'))
   - Config: Serves files from 'uploads' directory
   - Applied: Globally for /uploads path
   - Status: APPLICATION-LEVEL

2. CUSTOM MIDDLEWARES (Project-Created)
================================================================================

Custom middlewares are created specifically for this project's business logic.

FILE: authMiddleware.js
Location: WBD/backend/src/middlewares/authMiddleware.js

2.1 authenticateJWT()
    - Type: Authentication middleware
    - Purpose: Verify JWT token validity
    - Logic: Extracts Bearer token, verifies against JWT_SECRET
    - Returns: 401 if no token, 403 if invalid
    - Router-level Usage: Applied to protected routes
    - Example: Route /api/profile requires authentication

2.2 ensureDietitianAuthenticated()
    - Type: Role-based authorization middleware
    - Purpose: Ensure only dietitians can access specific routes
    - Logic: Verifies JWT token and checks if role === 'dietitian'
    - Returns: 401 if no token, 403 if not dietitian
    - Router-level Usage: Applied to dietitian-only endpoints
    - Example: Dietitian profile management routes

2.3 ensureOrganizationAuthenticated()
    - Type: Role-based authorization middleware
    - Purpose: Ensure only organizations can access specific routes
    - Logic: Verifies JWT and checks if role === 'organization'
    - Returns: 401 if no token, 403 if not organization
    - Router-level Usage: Applied to organization-only endpoints
    - Example: Organization management routes

2.4 ensureCorporatePartnerAuthenticated()
    - Type: Role-based authorization middleware
    - Purpose: Ensure only corporate partners can access routes
    - Logic: Verifies JWT and checks if role === 'corporatepartner'
    - Returns: 401 if no token, 403 if not corporate partner
    - Router-level Usage: Applied to corporate partner endpoints

2.5 validateDietitianObjectId()
    - Type: Validation middleware
    - Purpose: Validate MongoDB ObjectId format for dietitian IDs
    - Logic: Uses mongoose.Types.ObjectId.isValid()
    - Returns: 400 if invalid ID format
    - Router-level Usage: Applied before operations on specific IDs
    - Example: router.get('/files/:dietitianId/:field', validateDietitianObjectId, ...)

2.6 Additional Auth Validation Functions
    - validateOrganizationObjectId() - Same as above for organization IDs
    - validateCorporateObjectId() - Same as above for corporate partner IDs

FILE: securityMiddleware.js
Location: WBD/backend/src/middlewares/securityMiddleware.js

2.7 sanitizeInput()
    - Type: Input sanitization middleware
    - Purpose: Clean and escape user input to prevent XSS attacks
    - Logic: Escapes <, >, & characters in query params, body, and route params
    - Applied: Globally to all routes (app.use)
    - Status: APPLICATION-LEVEL
    - Order: After express.json() and express.urlencoded()
    - Security: Prevents script injection via user input

2.8 corsHeaders()
    - Type: CORS helper middleware (custom though cors is 3rd party)
    - Purpose: Set custom CORS headers in responses
    - Logic: Manually sets Access-Control headers
    - Applied: Could be used router-level if needed
    - Status: DEFINED but not actively used (cors() 3rd party is preferred)

FILE: loggerMiddleware.js
Location: WBD/backend/src/middlewares/loggerMiddleware.js

2.9 requestLogger (Morgan instance)
    - Type: Logging middleware
    - Purpose: Log all HTTP requests
    - Logic: Uses Morgan 'combined' format, writes to rotating file streams
    - Applied: Globally (app.use)
    - Status: APPLICATION-LEVEL
    - Output: Stored in logs/request.txt (rotates daily)
    - Details: Method, URL, status, response time, user agent

2.10 errorLogger()
     - Type: Error logging function
     - Purpose: Log application errors
     - Logic: Writes error details to rotating file stream
     - Applied: Called from errorHandler middleware
     - Status: UTILITY function
     - Output: Stored in logs/error.txt (rotates daily)
     - Details: Timestamp, error message, method, URL

FILE: errorMiddleware.js
Location: WBD/backend/src/middlewares/errorMiddleware.js

2.11 AppError (Class)
     - Type: Custom error class
     - Purpose: Standardize error structure across application
     - Properties: message, statusCode, status, isOperational, stack
     - Usage: throw new AppError('message', statusCode)
     - Example: throw new AppError('Invalid input', 400)

2.12 handleCastErrorDB()
     - Type: Error handler utility
     - Purpose: Handle MongoDB cast errors
     - Returns: AppError with 400 status
     - Triggered: When invalid MongoDB ObjectId provided

2.13 handleDuplicateFieldsDB()
     - Type: Error handler utility
     - Purpose: Handle duplicate field constraint violations
     - Returns: AppError with 400 status
     - Triggered: When unique field has duplicate value

2.14 handleValidationErrorDB()
     - Type: Error handler utility
     - Purpose: Handle MongoDB schema validation errors
     - Returns: AppError with 400 status
     - Triggered: When field validation fails

2.15 handleJWTError()
     - Type: Error handler utility
     - Purpose: Handle JWT verification failures
     - Returns: AppError with 401 status

2.16 handleJWTExpiredError()
     - Type: Error handler utility
     - Purpose: Handle expired JWT tokens
     - Returns: AppError with 401 status

2.17 sendErrorDev()
     - Type: Response handler
     - Purpose: Send detailed error response in development
     - Shows: Full error object, message, stack trace
     - Environment: NODE_ENV === 'development'

2.18 sendErrorProd()
     - Type: Response handler
     - Purpose: Send safe error response in production
     - Logic: Only shows message for operational errors
     - Hide: Error details to prevent information leakage

2.19 errorHandler()
     - Type: Global error handler middleware
     - Purpose: Catch and handle all application errors
     - Applied: app.use(errorHandler) at end of middleware chain
     - Status: APPLICATION-LEVEL (last middleware)
     - Logic: Routes errors through specific handlers based on type

2.20 notFoundHandler()
     - Type: 404 handler middleware
     - Purpose: Handle routes that don't exist
     - Applied: app.use(notFoundHandler) before errorHandler
     - Status: APPLICATION-LEVEL
     - Response: 404 status with "Route not found" message

2.21 catchAsync()
     - Type: Async error wrapper utility
     - Purpose: Wrap async route handlers to catch errors
     - Logic: Catches promise rejections and passes to next middleware
     - Usage: catchAsync(async (req, res, next) => { ... })
     - Benefit: Eliminates need for try/catch in async controllers

2.24 canCreateBlog()

2.25 isOrganization()

     - Logic: Checks if user role === 'organization'

FILE: cloudinaryMiddleware.js
Location: WBD/backend/src/middlewares/cloudinaryMiddleware.js

FILE: subscriptionMiddleware.js

2.30 checkBlogLimit()

3. 3RD-PARTY MIDDLEWARES (External Packages)
================================================================================

3rd-party middlewares come from npm packages to provide standard functionality.

3.1 helmet() [Package: helmet v8.1.0]
    - Purpose: Set security HTTP headers
    - Type: Security middleware (3rd party)
    - Import: const helmet = require('helmet')
    - Usage: app.use(helmet())
    - Applied: Globally (APPLICATION-LEVEL)
    - Location: Created in securityMiddleware.js
    - Headers Set: X-Frame-Options, X-Content-Type-Options, Strict-Transport-Security
    - Order: Very first middleware
    - Benefit: Protects against common web vulnerabilities

3.2 rateLimiter [Package: express-rate-limit v8.2.1]
    - Purpose: Limit requests per IP to prevent abuse
    - Type: Rate limiting middleware (3rd party)
    - Window: 15 minutes
    - Max Requests: 100,000 general, 10,000 for auth endpoints
    - Applied: Globally (APPLICATION-LEVEL)
    - Location: Created in securityMiddleware.js
    - Skip Rules: /api/health endpoint skips rate limit
    - Status Headers: Returns X-RateLimit-* headers
    - Order: Applied after helmet

3.3 authRateLimiter [Package: express-rate-limit v8.2.1]
    - Purpose: Stricter rate limiting for auth attempts
    - Type: Rate limiting middleware (3rd party)
    - Window: 15 minutes
    - Max Requests: 10,000 per IP
    - Applied: Router-level on auth routes
    - Location: Created in securityMiddleware.js
    - Skip Failed: skipSuccessfulRequests = true (rate limit only failures)
    - Benefit: Prevents brute force attacks

3.4 cors() [Package: cors v2.8.5]
    - Purpose: Enable Cross-Origin Resource Sharing
    - Type: CORS middleware (3rd party)
    - Import: const cors = require('cors')
    - Usage: app.use(cors())
    - Applied: Globally (APPLICATION-LEVEL)
    - Location: server.js (line 46)
    - Config: Defaults allow all origins (* by default)
    - Order: Applied after rateLimiter, before express.json()
    - Benefit: Frontend on different domain can call backend API

3.5 morgan() [Package: morgan v1.10.0]
    - Purpose: HTTP request logging
    - Type: Logging middleware (3rd party but used in custom requestLogger)
    - Import: const morgan = require('morgan')
    - Format: 'combined' format (Apache combined log format)
    - Applied: Globally as custom requestLogger (APPLICATION-LEVEL)
    - Output: Rotating file stream - logs/request.txt
    - Location: loggerMiddleware.js
    - Details: Method, URL, status, bytes sent, user-agent, response time

3.6 multer [Package: multer v2.0.2]
    - Purpose: Handle multipart/form-data (file uploads)
    - Type: File upload middleware (3rd party)
    - Storage: Memory storage (buffer)
    - Applied: Router-level on specific upload routes
    - Locations: uploadMiddleware.js and cloudinaryMiddleware.js
    - Instances: 
      * upload (uploadMiddleware.js) - 20MB limit, all file types
      * uploadBlogImage (cloudinaryMiddleware.js) - 5MB limit, images only

3.7 cloudinary [Package: cloudinary v2.8.0]
    - Purpose: Cloud storage and image transformation
    - Type: External service middleware (3rd party)
    - Config: API key, secret, cloud name from env
    - Usage: cloudinary.uploader.upload_stream() called in uploadToCloudinary
    - Applied: Router-level in blog routes
    - Transformations: Resize, quality optimization, auto format
    - Output: Cloudinary URL attached to req.cloudinaryResult

3.8 rotating-file-stream [Package: rfs v3.2.8]
    - Purpose: Rotate log files to prevent them from getting too large
    - Type: Logging utility (3rd party)
    - Import: const rfs = require('rotating-file-stream')
    - Config: Rotates daily, keeps 30 days of logs
    - Applied: In loggerMiddleware.js for request and error logs
    - Files: logs/request.txt and logs/error.txt
    - Benefit: Prevents disk space issues with ever-growing log files

3.9 jsonwebtoken [Package: jsonwebtoken v9.0.2]
    - Purpose: Create and verify JWT tokens
    - Type: Authentication utility (3rd party)
    - Import: const jwt = require('jsonwebtoken')
    - Usage: jwt.sign() to create tokens, jwt.verify() to verify
    - Applied: In authMiddleware, blogMiddleware, statusMiddleware
    - Secret: JWT_SECRET from environment variables
    - Benefit: Stateless authentication

3.10 mongoose [Package: mongoose v8.18.1]
     - Purpose: MongoDB object modeling
     - Type: Database middleware (3rd party)
     - Import: const mongoose = require('mongoose')
     - Usage: ObjectId validation, schema validation
     - Applied: In validators like validateDietitianObjectId
     - Benefit: Type safety and query building for MongoDB

4. APPLICATION-LEVEL MIDDLEWARES
================================================================================

Application-level middlewares are applied globally to ALL routes using app.use()
Location: server.js (line 31 onwards)

AML-1: Helmet Security Headers
       app.use(helmetMiddleware)
       Order: 1 (First)
       Package: 3rd-party
       Protection: Sets security headers

AML-2: Rate Limiting
       app.use(rateLimiter)
       Order: 2
       Package: 3rd-party
       Config: 100,000 requests/15min general, skips /api/health

AML-3: Request Logging
       app.use(requestLogger)
       Order: 3
       Package: 3rd-party (morgan)
       Output: logs/request.txt

AML-4: CORS Support
       app.use(cors())
       Order: 4
       Package: 3rd-party
       Config: Allows cross-origin requests

AML-5: JSON Body Parser
       app.use(express.json({ limit: '10mb' }))
       Order: 5
       Package: Built-in
       Config: 10MB size limit

AML-6: URL-Encoded Body Parser
       app.use(express.urlencoded({ limit: '10mb', extended: true }))
       Order: 6
       Package: Built-in
       Config: 10MB size limit, extended parsing

AML-7: Input Sanitization
       app.use(sanitizeInput)
       Order: 7
       Package: Custom
       Protection: Escapes <, >, & in all user input

AML-8: Static File Server
       app.use('/uploads', express.static('uploads'))
       Order: 8
       Package: Built-in
       Path: /uploads serves files from uploads/ directory

AML-9: Global 404 Handler
       app.use(notFoundHandler)
       Order: Before errorHandler
       Package: Custom
       Response: 404 for undefined routes

AML-10: Global Error Handler
        app.use(errorHandler)
        Order: LAST
        Package: Custom
        Catches: All errors thrown or passed via next()

MIDDLEWARE EXECUTION FLOW
================================================================================

Request Flow for Authenticated Blog Creation:

1. Client sends POST /api/blogs with JWT token
2. Express receives request
3. Application-level middlewares execute in order:
   a. helmet() - Sets security headers
   b. rateLimiter - Checks rate limit
   c. requestLogger - Logs request
   d. cors() - Checks origin
   e. express.json() - Parses JSON body
   f. express.urlencoded() - Parses form data
   g. sanitizeInput - Cleans input
4. Router matches /blogs POST
5. Router-level middlewares execute:
   a. verifyBlogAuth() - Verify JWT, attach user
   b. canCreateBlog() - Check role can create
   c. checkBlogLimit() - Check subscription limits
   d. uploadBlogImage.single() - Parse image upload
   e. uploadToCloudinary() - Upload to Cloudinary
6. Controller executes (blogController.createBlog)
7. Response sent
8. Error handler catches any errors
